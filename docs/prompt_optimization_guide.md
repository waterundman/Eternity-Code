# Prompt优化论：元迭代Agent的设计原则

> 适用于：学习评估Agent项目（Node.js + GLM API）
> 核心命题：**Prompt质量不是越详尽越好，而是存在甜点区间。**

---

## 一、核心理论

### 1.1 甜点密度原则

Prompt的指令密度与输出质量之间是一个倒U型关系：

```
输出质量
    ▲
    │         ●  ← 甜点区间
    │       ●   ●
    │     ●       ●
    │   ●           ●
    │ ●               ●
    └─────────────────────→ 指令密度
      稀疏               堆砌
```

**过稀疏**：模型缺乏约束，方向漂移  
**甜点区间**：核心任务清晰 + 必要约束 + 足够留白  
**过堆砌**：模型注意力被分散，内部冲突增多，推理空间被压缩

### 1.2 第一次输出偏差原理

模型的第一次输出通常质量最高，原因：

- Context最干净，信噪比最高
- 没有被前一次"糟糕输出"锚定
- 不受"修改请求"隐含的负反馈信号干扰

**推论**：与其反复让模型修改，不如把修改动作**前移到prompt生成阶段**。

## 二、架构设计原则

### 2.1 三层架构

```
┌─────────────────────────────────────┐
│  Layer 1: 元层（Meta Layer）         │
│  负责生成子prompt                    │
│  内化甜点密度原则                    │
└──────────────────┬──────────────────┘
                   │
┌──────────────────▼──────────────────┐
│  Layer 2: 优化Pass（Optimization）   │
│  剪枝冗余约束                        │
│  检测指令冲突                        │
│  输出标准化                          │
└──────────────────┬──────────────────┘
                   │
┌──────────────────▼──────────────────┐
│  Layer 3: 执行层（Execution Layer）  │
│  单次生成，不做多轮迭代修改           │
│  永远面对干净的第一次context          │
└─────────────────────────────────────┘
```

### 2.2 元层的Prompt生成规则

元层在生成子prompt时，必须遵守以下约束：

```
✅ DO
- 核心任务用1-2句话说清楚
- 约束只写影响输出质量的必要项
- 给模型留出填充细节的自由度
- 输出格式只规定结构，不规定措辞

❌ DON'T
- 穷举所有可能的细节
- 写互相冲突的约束（如"简洁"+"详细举例"）
- 预设模型的推理路径
- 把每一个输出字段都规定死
```

## 三、优化Pass的实现

### 3.1 触发条件

每次元层生成子prompt后，优化Pass自动运行。

### 3.2 检查项

| 检查项 | 方法 | 处理 |
|--------|------|------|
| 指令密度 | token数 / 核心意图数 | 超过阈值则剪枝 |
| 冲突检测 | 关键词对匹配（简洁vs详细、快vs慢等） | 标记冲突，保留优先级高的 |
| 冗余约束 | 语义相似度检测 | 合并或删除 |
| 留白检查 | 是否有over-specify的字段 | 释放非必要约束 |

### 3.3 伪代码示例（Node.js）

```javascript
async function optimizePrompt(rawPrompt) {
  const densityScore = calcDensity(rawPrompt); // token/intent ratio
  
  if (densityScore > DENSITY_THRESHOLD) {
    rawPrompt = await pruneRedundancy(rawPrompt);
  }
  
  const conflicts = detectConflicts(rawPrompt);
  if (conflicts.length > 0) {
    rawPrompt = resolveConflicts(rawPrompt, conflicts);
  }
  
  return rawPrompt;
}
```

## 四、反馈环设计

### 4.1 信号来源

用户对卡片的评分 → 反推对应prompt模板的质量

```
卡片评分（用户行为）
       ↓
聚合为prompt模板评分（去除个体噪音）
       ↓
更新元层对"好prompt"的判断
       ↓
下一轮生成更接近甜点
```

### 4.2 噪音处理

单次评分信号噪音大，**不能直接用于反推**。

推荐处理方式：

- 使用**同一模板在N次以上的均值**作为信号（N ≥ 5）
- 区分噪音类型：
  - 内容噪音（用户不喜欢这个话题）→ 过滤
  - 结构噪音（卡片顺序/时机问题）→ 单独分析
  - Prompt质量信号（多次相同低分）→ 触发优化

### 4.3 可学习参数

随系统运行，以下参数应能自动校准：

- `DENSITY_THRESHOLD`：最优指令密度阈值（按任务类型细分）
- 冲突词对表：高频冲突词组的优先级规则
- 留白策略：哪些字段应该由模型自由填充

## 五、迭代路线

### Phase 1：写死原则（当前阶段）
- 将甜点原则硬编码进元层的system prompt
- 手动维护冲突词对表
- 评分系统正常收集数据，不做自动反馈

### Phase 2：规则优化Pass
- 实现density check和conflict detection
- 优化Pass作为独立模块接入pipeline
- 开始用评分均值评估不同prompt模板

### Phase 3：自适应校准
- 用积累的评分数据驱动参数自动更新
- 按用户类型/任务类型做细粒度的甜点校准
- 对比A/B：优化前vs优化后的评分分布

---

## 六、一句话原则（随时可查）

> **把"修改"前移到prompt生成阶段，让执行层永远在第一次、干净的context下工作。**
